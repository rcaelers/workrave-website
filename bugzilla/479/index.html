Description
Rob Caelers  Sep 8 2005 22:30:44 UTC
We get quite some mail from people who want to increase the amount of idle time
it takes before workrave stops the timers. Currently, it is set to 5s. When
reading a document on screen, 5s is way to short. People suggest a timeout of 2
minutes.

I suggest to add a new operation mode (next to normal, suspended and quiet)
calling 'reading'. This mode has an increaded idle-timeout setting (default 2
minutes). We might even make the normal and reading idle-timeout configurable...

Comments?
Comment 1
Kees-Jan Dijkzeul  Sep 9 2005 09:08:41 UTC
It's an interesting idea. I'm not sure I'd implement it as a "mode" a la normal,
quiet and suspended. For one, they are completely different things (one changes
the amount of warnings, one changes timeout values), secondly they are
orthogonal (i.e. you could combine reading mode with each of the other three)
and lastly the "mode" menu is hidden way too well. You'd want to have reading
mode more readily accessible.

I'm a bit concerned how such a long timeout would interact with other features.
For one, I'd have a pretty tough time trying to have a natural microbreak.

I'm also a bit concerned how this feature would impact the "feel" of workrave.
Having such a long timeout will make the user feel less in control of whether he
is active or not. We might need "microbreak now" and "reset microbreak timer"
buttons.

Either way, the only way to know the impact of this change would be to try. But
then we need someone volonteering to test this. I'm not doing a lot of on-screen
reading :-)

And... Eh... Am I to late to request webcam support?? ;-)
Comment 2
GS  Aug 23 2006 18:58:39 UTC
Reading Mode:

I feel with others on the importance of the reading mode. When reading a browser page or pdf document on the screen, it is desirable to have micropauses. But workrave thinks that you are taking a rest and stops the timers. So, it happens that you read the screen for half an hour or more without a pause.

As the previous comments indicate, it is difficult for workrave to determine whether a person is reading or is really idle. One solution is to give a reading-break prompt anyway after a longer duration (than microbreak) and remove that prompt if there is mouse or key board activity. The reason is that the "time between breaks" for reading-break can be longer than a micro-break. An idle time without user input can be considered as potential reading time for giving this reading-break.

The reading-break is a micro-break but should not force it just in case the user has returned from a real rest-break and wants to resume working.

As Kees-Jan says, the only way of seeing how this feels is to try it. I am willing to test it out (or actually program the feature if I get help on how to dig into code). In any case thanks for this nice piece of software. I was using a proprietary RSI software, but switched to workrave because the linux lincense was double the price of windows license.

Kees-Jan Dijkzeul  Sep 25 2006 22:51:48 UTC
(In reply to comment #3)

Actually, I'm quite happy with the current situation. It works fine for almost all tasks except for reading. Let me elaborate a little...

The purpose of a microbreak is not to get you away from the computer. That's wat restbreaks are for. The purpose of a microbreak is to make you relax for a few seconds in between periods of working. 

In fact you are fortunate that your work has such "natural" periods. A datatypist, for example, would theoretically be able to type for hours on end without a "natural" need to break (other than logistics such as lunch room or bath room visits).

For best results, you should get used to taking a more relaxed position while thinking. That way, Workrave is correct in assuming that you are not physically active. Of course, thinking counts as work too, just not as computerwork :-)

I originally asked for webcam support, such that Workrave would know whether or not I was reading from screen, and thus be able to prompt for microbreaks. Your post makes me think of this again. I think workrave would have a hard time distinguishing me reading a screen from me taking a natural microbreak. Maybe, asking for webcam support was not a good idea after all. At least, I've got some more thinking to do.

I use Workrave primarily to relieve computer-caused eye strain. I need to take a break after half an hour of work, however I appreciate Workrave's ability to naturally detect when I've taken a break (suspend on inactive).

Adding webcam support is overkill and risks seriously bloating the program.

I have a simpler suggestion. When I am reading I often notice that Workrave has stopped counting "work time" and has started raising the green "break time" meter. If I wave the mouse when this happens, Workrave realizes that I'm not actually taking a break and the green bar disappears.

This is good but for reading, not enough. If I have *not* been taking a break during the inactive period, it is probably safe to assume that I have been looking at the computer screen and thus (for eye-strain users) "working". What Workrave can do is take the break time (green) and add it to my work timer.

Adding a toggle button in the options menu that would enable this simple functionality would be ideal.
